# 誰でも使える形にする：ウェブアプリ vs ネイティブアプリ

ローカルではなく「本格的なウェブアプリ」または「ネイティブアプリ」にする場合の選択肢と、必要な変更をまとめました。

---

## 1. 本格的なウェブアプリ（ブラウザでURLを開くだけ）

**イメージ**: サービスを1回デプロイしておき、利用者はブラウザで https://あなたのサービス.com を開くだけで使える。

### 必要な主な変更

| 今（ローカル） | ウェブアプリ化で必要になること |
|----------------|--------------------------------|
| 設定・キュー・トークンをフォルダ内のファイルで保存 | **データベース**で保存（ユーザーごとにキュー・アカウントを分離） |
| 誰でも同じPCの画面を見る | **ユーザー認証**（ログイン）で「自分のキュー・自分のアカウント」だけ表示 |
| OAuth が「このPCのブラウザで開く」 | **OAuth の「ウェブアプリケーション」** を使い、認証後に**サーバーへリダイレクト**してトークンを保存 |
| 動画はPC内のパスを指定 | **動画をサーバーにアップロード**してもらう → サーバーから YouTube API でアップロード |

### 技術の候補

- **バックエンド**: FastAPI または Flask（API + OAuth コールバック + 動画の一時保存）
- **データベース**: PostgreSQL または SQLite（ユーザー・キュー・トークン（暗号化）を保存）
- **認証**: メール+パスワード + セッション、または Firebase Auth / Auth0 など
- **フロント**: 現在の Streamlit を「ログイン付き」で使う、または React / Vue で新規作成
- **デプロイ**: Railway, Render, Fly.io, 自前 VPS + Docker など

### 流れ（ユーザー目線）

1. サイトにアクセス → ログイン（または会員登録）
2. 「YouTube アカウントを連携」→ Google の認証画面 → 許可 → サーバーがトークンを保存
3. キューに「動画ファイルをアップロード」してタイトル等を入力
4. 「アップロード実行」→ サーバーが YouTube にアップロード

### コスト・注意点

- **サーバー代**: 月数百円〜（Railway/Render の無料枠もあり、トラフィック次第）
- **ストレージ**: 動画を一時保存するため、容量と転送量が増えるとコスト増
- **GCP**: OAuth の「ウェブアプリケーション」用に**リダイレクト URI** を登録する必要あり
- **セキュリティ**: トークンは暗号化してDBに保存、HTTPS 必須

---

## 2. ネイティブアプリ（インストールして使う）

**イメージ**: .exe（Windows）や .app（Mac）を配布し、ユーザーがダウンロードして起動。Python を入れなくても動く。

### 必要な主な変更

- **配布形式**: 今の Python アプリを **PyInstaller** などで1つの実行ファイル（またはフォルダ）にまとめる
- 起動すると **ローカルで Web サーバー（Streamlit）が立ち上がり**、ブラウザで `http://localhost:8501` が開く
- データは今と同様、**そのPCのフォルダ内**（config、accounts、キュー）に保存

### メリット・デメリット

- **メリット**: サーバー不要、実装変更が少ない、「Python を入れずに使える」で配布しやすい
- **デメリット**: 依然として「各自のPCで動かす」ので、「URLを共有して誰でも」という使い方にはならない

### 技術の候補

- **PyInstaller**: `pyinstaller run_app.py` のようにして exe 化。Python ランタイムを同梱できる
- **注意**: Streamlit は内部でサーバーを立てるため、PyInstaller のオプション（隠しインポートなど）の調整が必要になることがある

---

## 3. 比較とおすすめ

| 目的 | 向いている形 |
|------|----------------|
| **「URLを教えるだけで、誰でもブラウザから使える」にしたい** | 本格的な**ウェブアプリ**（認証・DB・OAuth ウェブ用・動画アップロード） |
| **「Python を入れずに、配布用の exe で使ってほしい」** | **ネイティブアプリ**（PyInstaller で exe 化） |
| **「とりあえずコストをかけずに、今の仕組みのまま広げたい」** | 現状のローカル版 + はじめて使う人向けドキュメント（README・はじめて使う人向け.md）のまま配布 |

---

## 4. ウェブアプリ化を進める場合のステップ案

1. **設計**
   - ユーザー認証の方式（メール/パスワード or Google ログインなど）
   - DB スキーマ（users, accounts（YouTube トークン）, queue テーブル）
   - OAuth リダイレクト URL の設計（例: `https://あなたのドメイン/api/youtube/callback`）

2. **バックエンド**
   - FastAPI で API を用意（ログイン、キュー一覧・追加・実行、YouTube 連携の開始・コールバック）
   - 動画アップロード用のエンドポイント（マルチパートで受け取り、一時ファイルとして保存）
   - キュー実行は「非同期ジョブ」として実行（長時間になるため）

3. **フロント**
   - 現在の Streamlit を「ログイン必須」にしてバックエンド API を呼ぶ形に変更、または
   - React/Vue で新規画面を作り、見た目と運用を本格化

4. **デプロイ**
   - Railway / Render などにデプロイ
   - GCP の OAuth クライアントに「ウェブアプリケーション」を追加し、リダイレクト URI を登録

---

## 5. ネイティブアプリ化を進める場合のステップ案

1. **PyInstaller の導入**
   ```bash
   pip install pyinstaller
   ```

2. **spec ファイルで Streamlit を同梱**
   - `run_app.py` をエントリポイントに、Streamlit と依存ライブラリをまとめる
   - 隠しインポート（streamlit 関連）を必要に応じて追加

3. **ビルド**
   ```bash
   pyinstaller （作成した .spec ファイル）
   ```

4. **配布**
   - 生成された exe（またはフォルダ）を ZIP で配布
   - 「初回は config と client_secrets の配置が必要」などは、同梱の README で案内

---

どちらを優先するか（「まずは exe で配布」か「いずれウェブで誰でも」か）が決まれば、その方向に合わせて次の具体的なタスク（API 一覧、DB スキーマ、または PyInstaller の spec 例）を細かく書けます。
